คู่มือนี้จะแนะนำขั้นตอนการสร้างบทบาท RBAC แบบกำหนดเองใน F5 Distributed Cloud
ที่ให้สิทธิ์ในการจัดการใบรับรอง SSL/TLS และสายใบรับรอง (certificate chains)
ที่ใช้โดย HTTP Load Balancers สำหรับแนวทางผ่าน UI สามารถดูได้ที่
[คู่มือคอนโซลแบบง่าย](./custom-role-ui)

## วิธีการทำงานของ F5 XC RBAC

F5 XC ใช้โมเดลสิทธิ์แบบสามระดับ:

```
api_group_element  (อ่านอย่างเดียว, กำหนดโดยระบบ)
       │
       ▼
   api_group       (อ่านอย่างเดียว, กำหนดโดยระบบ)
       │
       ▼
     role           (CRUD เต็มรูปแบบ — นี่คือสิ่งที่คุณสร้าง)
```

1. **`api_group_element`** — กำหนดเส้นทาง regex และ HTTP methods
   (เช่น `POST /api/config/.*/certificates`) จัดการโดยระบบ; คุณไม่สามารถ
   สร้าง แก้ไข หรือลบสิ่งเหล่านี้ได้

2. **`api_group`** — คอลเลกชันที่มีชื่อของการอ้างอิง `api_group_element`
   จัดระเบียบตามพื้นที่การทำงาน จัดการโดยระบบ; คุณไม่สามารถสร้าง แก้ไข หรือ
   ลบสิ่งเหล่านี้ได้ กลุ่มใช้แบบแผนการตั้งชื่อเช่น `ves-io-proxy-read` หรือ
   `f5xc-waap-standard-admin`

3. **`role`** — อ้างอิงชื่อ `api_group` ในอาร์เรย์ **นี่คือระดับเดียว
   ที่คุณสร้าง** ใช้ endpoint ของบทบาทแบบกำหนดเองเพื่อแนบ
   `api_groups` เฉพาะเข้ากับบทบาท

การดำเนินการเกี่ยวกับใบรับรองสำหรับ HTTP Load Balancers ถูกรวมอยู่ใน
api_groups `ves-io-proxy-*` ร่วมกับทรัพยากรที่เกี่ยวข้องกับ proxy อื่นๆ (HTTP Load
Balancers, routes, origins ฯลฯ) ไม่มี api_groups เฉพาะสำหรับใบรับรองเพียงอย่างเดียว — แพลตฟอร์มจัดระเบียบสิทธิ์ตามพื้นที่การทำงาน ไม่ใช่ตามประเภททรัพยากรแต่ละรายการ

## ขอบเขตและข้อจำกัด

api_groups `ves-io-proxy-read` และ `ves-io-proxy-write` ให้สิทธิ์เข้าถึง
**ทรัพยากรที่เกี่ยวข้องกับ proxy ทั้งหมด** ไม่ใช่เฉพาะใบรับรอง ผู้ใช้ที่ได้รับมอบหมายบทบาทนี้
สามารถจัดการ HTTP Load Balancers, origins, routes และอ็อบเจกต์ในขอบเขต proxy
อื่นๆ ได้เช่นกัน

RBAC เฉพาะใบรับรองไม่สามารถทำได้บน F5 XC ทั้ง `api_group` และ
`api_group_element` เป็นอ็อบเจกต์ที่กำหนดโดยระบบและอ่านอย่างเดียว — API เปิดเผย
เฉพาะ GET endpoints สำหรับสิ่งเหล่านี้ บทบาทสามารถอ้างอิงชื่อ `api_group`
ที่มีอยู่เท่านั้น และไม่มี `api_group` ที่กำหนดขอบเขตเฉพาะสำหรับใบรับรอง

**การบรรเทา — การแยก namespace:** สร้าง namespace เฉพาะสำหรับ
ทรัพยากรใบรับรองและมอบหมายบทบาท `example-cert-admin` ที่กำหนดขอบเขตเฉพาะ
namespace นั้นเท่านั้น ผู้ใช้ยังคงมีสิทธิ์ระดับ proxy แต่ไม่สามารถเข้าถึง
ทรัพยากรใน namespace อื่นๆ ซึ่งจำกัดผลกระทบจากขอบเขต api_group ที่กว้างกว่า

## ข้อกำหนดเบื้องต้น

- API token ของ F5 XC ที่มีสิทธิ์ admin
- ติดตั้ง `curl` และ `jq` แล้ว

ตั้งค่าตัวแปรสภาพแวดล้อมของคุณ:

```bash
export F5XC_TENANT="your-tenant"
export F5XC_API_URL="https://${F5XC_TENANT}.console.ves.volterra.io"
export F5XC_API_TOKEN="your-api-token"
```

## ขั้นตอนที่ 1: ค้นหา Proxy API Groups

แสดงรายการอ็อบเจกต์ `api_group` ที่กำหนดโดยระบบทั้งหมดและกรองรายการที่เกี่ยวข้องกับ proxy
(ซึ่งรวมถึงสิทธิ์ใบรับรอง):

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/shared/api_groups" \
  | jq '[.items[] | select(.name | test("ves-io-proxy")) | .name] | sort'
```

ผลลัพธ์ที่คาดหวัง:

```json
[
  "ves-io-proxy-app-firewall-read",
  "ves-io-proxy-app-firewall-write",
  "ves-io-proxy-monitor-read",
  "ves-io-proxy-monitor-write",
  "ves-io-proxy-read",
  "ves-io-proxy-security-read",
  "ves-io-proxy-security-write",
  "ves-io-proxy-waf-read",
  "ves-io-proxy-waf-write",
  "ves-io-proxy-write"
]
```

สำหรับการจัดการใบรับรอง กลุ่มที่เกี่ยวข้องคือ:

| API Group | สิทธิ์ที่ให้ |
| --------- | ------ |
| `ves-io-proxy-read` | สิทธิ์อ่านทรัพยากร proxy รวมถึงใบรับรอง |
| `ves-io-proxy-write` | สิทธิ์เขียนทรัพยากร proxy รวมถึงใบรับรอง |

:::note
กลุ่มเหล่านี้ครอบคลุมทรัพยากรที่เกี่ยวข้องกับ proxy ทั้งหมด (HTTP Load Balancers, origins,
routes, ใบรับรอง, สายใบรับรอง ฯลฯ) ไม่ใช่เฉพาะใบรับรอง F5 XC
จัดระเบียบสิทธิ์ตามพื้นที่การทำงาน หากคุณต้องการตรวจสอบว่ากลุ่มให้สิทธิ์
เส้นทาง API เฉพาะใดบ้าง ให้ตรวจสอบ endpoint `api_group_elements` ที่อธิบายไว้ใน
ส่วนอ้างอิง API
:::

## ขั้นตอนที่ 2: สร้างบทบาทแบบกำหนดเอง

ใช้ endpoint สำหรับสร้างบทบาทแบบกำหนดเอง อาร์เรย์ `api_groups` อ้างอิง
ชื่อ api_group ที่กำหนดโดยระบบซึ่งค้นพบในขั้นตอนที่ 1

สร้างบทบาทแบบกำหนดเองที่มีสิทธิ์อ่านและเขียนทรัพยากร proxy (รวมถึง
ใบรับรองและสายใบรับรอง):

```bash
curl -s -X POST \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  -H "Content-Type: application/json" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles" \
  -d '{
    "metadata": {
      "name": "example-cert-admin",
      "namespace": "system"
    },
    "spec": {},
    "api_groups": [
      "ves-io-proxy-read",
      "ves-io-proxy-write"
    ]
  }'
```

## ขั้นตอนที่ 3: ตรวจสอบบทบาท

ดึงข้อมูลบทบาทโดยใช้ custom GET endpoint เพื่อยืนยันว่า `api_groups`
ถูกแนบแล้ว:

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles/example-cert-admin" \
  | jq '{name: .object.metadata.name, api_groups: .api_groups}'
```

ผลลัพธ์ที่คาดหวัง:

```json
{
  "name": "example-cert-admin",
  "api_groups": [
    "ves-io-proxy-read",
    "ves-io-proxy-write"
  ]
}
```

## ขั้นตอนที่ 4: อัปเดตบทบาท (ไม่บังคับ)

เพื่อเพิ่มหรือลบ `api_groups` จากบทบาทที่มีอยู่ ให้ใช้ custom PUT
endpoint:

```bash
curl -s -X PUT \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  -H "Content-Type: application/json" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles/example-cert-admin" \
  -d '{
    "name": "example-cert-admin",
    "namespace": "system",
    "spec": {},
    "api_groups": [
      "ves-io-proxy-read"
    ]
  }'
```

ตัวอย่างนี้ลดระดับบทบาทเป็นสิทธิ์อ่านอย่างเดียวโดยลบ
`ves-io-proxy-write` ออก

## ขั้นตอนที่ 5: ลบบทบาท (ไม่บังคับ)

การลบใช้ endpoint บทบาทมาตรฐาน:

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/system/roles/example-cert-admin"
```

## อ้างอิง API

### Endpoint ของบทบาท (แบบกำหนดเอง — รวม `api_groups`)

บทบาทจะอยู่ใน namespace `system` เสมอ

| Method | เส้นทาง | คำอธิบาย |
| ------ | ---- | ----------- |
| POST | `/api/web/custom/namespaces/system/roles` | สร้างบทบาทแบบกำหนดเอง |
| GET | `/api/web/custom/namespaces/system/roles` | แสดงรายการบทบาทแบบกำหนดเอง |
| GET | `/api/web/custom/namespaces/system/roles/{name}` | ดึงข้อมูลบทบาทแบบกำหนดเอง |
| PUT | `/api/web/custom/namespaces/system/roles/{name}` | แทนที่บทบาทแบบกำหนดเอง |

### Endpoint ของบทบาท (มาตรฐาน)

| Method | เส้นทาง | คำอธิบาย |
| ------ | ---- | ----------- |
| DELETE | `/api/web/namespaces/system/roles/{name}` | ลบบทบาท |
| GET | `/api/web/namespaces/system/roles/{name}` | ดึงข้อมูลบทบาท |
| GET | `/api/web/namespaces/system/roles` | แสดงรายการบทบาท |

### Endpoint สำหรับค้นหา (อ่านอย่างเดียว)

การค้นหาใช้ namespace `shared`

| Method | เส้นทาง | คำอธิบาย |
| ------ | ---- | ----------- |
| GET | `/api/web/namespaces/shared/api_groups` | แสดงรายการ api_groups ทั้งหมด |
| GET | `/api/web/namespaces/shared/api_group_elements` | แสดงรายการ api_group_elements ทั้งหมด |

### Endpoint ของใบรับรอง (สิ่งที่บทบาทให้สิทธิ์เข้าถึง)

แทนที่ `{namespace}` ด้วย namespace ที่ใบรับรองของคุณถูกจัดการ

| Method | เส้นทาง | คำอธิบาย |
| ------ | ---- | ----------- |
| POST | `/api/config/namespaces/{namespace}/certificates` | สร้างใบรับรอง |
| GET | `/api/config/namespaces/{namespace}/certificates` | แสดงรายการใบรับรอง |
| GET | `/api/config/namespaces/{namespace}/certificates/{name}` | ดึงข้อมูลใบรับรอง |
| PUT | `/api/config/namespaces/{namespace}/certificates/{name}` | แทนที่ใบรับรอง |
| DELETE | `/api/config/namespaces/{namespace}/certificates/{name}` | ลบใบรับรอง |
| POST | `/api/config/namespaces/{namespace}/certificate_chains` | สร้างสายใบรับรอง |
| GET | `/api/config/namespaces/{namespace}/certificate_chains` | แสดงรายการสายใบรับรอง |
| GET | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | ดึงข้อมูลสายใบรับรอง |
| PUT | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | แทนที่สายใบรับรอง |
| DELETE | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | ลบสายใบรับรอง |

## เหตุใด `api_group` และ `api_group_element` จึงเป็นแบบอ่านอย่างเดียว

อ็อบเจกต์เหล่านี้ถูก**กำหนดโดยระบบ**ของ F5 XC แพลตฟอร์มสร้าง
รายการ `api_group_element` ล่วงหน้าสำหรับทุกเส้นทาง API และจัดกลุ่มเป็น
อ็อบเจกต์ `api_group` ที่จัดระเบียบตามพื้นที่การทำงาน

คุณไม่จำเป็นต้องสร้างสิ่งเหล่านี้ — มันมีอยู่แล้วสำหรับทุกทรัพยากรใน
ระบบ หน้าที่ของคุณคือ**ค้นหา**ชื่อกลุ่มที่เกี่ยวข้อง (ขั้นตอนที่ 1) และ
**อ้างอิง**ในบทบาทแบบกำหนดเองของคุณ (ขั้นตอนที่ 2)

เพื่อดูว่า api_group ครอบคลุมเส้นทาง API แต่ละรายการใดบ้าง ให้แสดงรายการ
endpoint `api_group_elements` และค้นหาองค์ประกอบที่เกี่ยวข้องกับทรัพยากรของคุณ:

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/shared/api_group_elements" \
  | jq '[.items[] | select(.name | test("ves-io-schema-certificate")) | .name]'
```
