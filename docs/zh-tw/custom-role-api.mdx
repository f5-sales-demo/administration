本指南將逐步說明如何在 F5 Distributed Cloud 中建立自訂 RBAC 角色，授予管理 HTTP Load Balancer 所使用的 SSL/TLS 憑證和憑證鏈的權限。如需基於 UI 的方法，請參閱[簡化的主控台指南](./custom-role-ui)。

## F5 XC RBAC 運作方式

F5 XC 使用三層權限模型：

```
api_group_element  (read-only, system-defined)
       │
       ▼
   api_group       (read-only, system-defined)
       │
       ▼
     role           (full CRUD — this is what you create)
```

1. **`api_group_element`** — 定義正規表示式路徑和 HTTP 方法（例如 `POST /api/config/.*/certificates`）。由系統管理；您無法建立、修改或刪除這些物件。

2. **`api_group`** — `api_group_element` 參考的具名集合，按功能區域組織。由系統管理；您無法建立、修改或刪除這些物件。群組使用如 `ves-io-proxy-read` 或 `f5xc-waap-standard-admin` 的命名慣例。

3. **`role`** — 在陣列中參考 `api_group` 名稱。**這是您唯一需要建立的層級。** 使用自訂角色端點將特定的 `api_groups` 附加到角色。

HTTP Load Balancer 的憑證操作與其他代理相關資源（HTTP Load Balancer、路由、來源等）一同包含在 `ves-io-proxy-*` api_groups 中。沒有獨立的僅限憑證的 api_groups — 平台按功能區域組織權限，而非按個別資源類型。

## 範圍與限制

`ves-io-proxy-read` 和 `ves-io-proxy-write` api_groups 授予對**所有代理相關資源**的存取權，而不僅僅是憑證。被指派此角色的使用者也可以管理 HTTP Load Balancer、來源、路由及其他代理範圍的物件。

在 F5 XC 上無法實現僅限憑證的 RBAC。`api_group` 和 `api_group_element` 都是系統定義的唯讀物件 — API 僅公開這些物件的 GET 端點。角色只能參考現有的 `api_group` 名稱，且沒有專門限定於憑證的 `api_group`。

**緩解措施 — 命名空間隔離：** 為憑證資源建立專用的命名空間，並將 `example-cert-admin` 角色僅指派到該命名空間的範圍內。使用者保留代理層級的權限，但無法存取其他命名空間中的資源，從而限制較廣泛 api_group 範圍的影響範圍。

## 先決條件

- 具有管理員權限的 F5 XC API 權杖
- 已安裝 `curl` 和 `jq`

設定您的環境變數：

```bash
export F5XC_TENANT="your-tenant"
export F5XC_API_URL="https://${F5XC_TENANT}.console.ves.volterra.io"
export F5XC_API_TOKEN="your-api-token"
```

## 步驟 1：探索代理 API 群組

列出所有系統定義的 `api_group` 物件，並篩選代理相關的項目（包含憑證權限）：

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/shared/api_groups" \
  | jq '[.items[] | select(.name | test("ves-io-proxy")) | .name] | sort'
```

預期輸出：

```json
[
  "ves-io-proxy-app-firewall-read",
  "ves-io-proxy-app-firewall-write",
  "ves-io-proxy-monitor-read",
  "ves-io-proxy-monitor-write",
  "ves-io-proxy-read",
  "ves-io-proxy-security-read",
  "ves-io-proxy-security-write",
  "ves-io-proxy-waf-read",
  "ves-io-proxy-waf-write",
  "ves-io-proxy-write"
]
```

對於憑證管理，相關的群組為：

| API 群組 | 授予權限 |
| --------- | ------ |
| `ves-io-proxy-read` | 代理資源（包含憑證）的讀取存取權 |
| `ves-io-proxy-write` | 代理資源（包含憑證）的寫入存取權 |

:::note
這些群組涵蓋所有代理相關資源（HTTP Load Balancer、來源、路由、憑證、憑證鏈等），而不僅僅是憑證。F5 XC 按功能區域組織權限。如果您需要驗證某個群組授予哪些特定 API 路徑，請查看 API 參考章節中描述的 `api_group_elements` 端點。
:::

## 步驟 2：建立自訂角色

使用自訂角色建立端點。`api_groups` 陣列參考在步驟 1 中探索到的系統定義 api_group 名稱。

建立具有代理資源（包含憑證和憑證鏈）讀寫存取權的自訂角色：

```bash
curl -s -X POST \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  -H "Content-Type: application/json" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles" \
  -d '{
    "metadata": {
      "name": "example-cert-admin",
      "namespace": "system"
    },
    "spec": {},
    "api_groups": [
      "ves-io-proxy-read",
      "ves-io-proxy-write"
    ]
  }'
```

## 步驟 3：驗證角色

使用自訂 GET 端點擷取角色，以確認 `api_groups` 已附加：

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles/example-cert-admin" \
  | jq '{name: .object.metadata.name, api_groups: .api_groups}'
```

預期輸出：

```json
{
  "name": "example-cert-admin",
  "api_groups": [
    "ves-io-proxy-read",
    "ves-io-proxy-write"
  ]
}
```

## 步驟 4：更新角色（選用）

若要從現有角色新增或移除 `api_groups`，請使用自訂 PUT 端點：

```bash
curl -s -X PUT \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  -H "Content-Type: application/json" \
  "${F5XC_API_URL}/api/web/custom/namespaces/system/roles/example-cert-admin" \
  -d '{
    "name": "example-cert-admin",
    "namespace": "system",
    "spec": {},
    "api_groups": [
      "ves-io-proxy-read"
    ]
  }'
```

此範例透過移除 `ves-io-proxy-write` 將角色降級為唯讀存取權。

## 步驟 5：刪除角色（選用）

刪除使用標準角色端點：

```bash
curl -s -X DELETE \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/system/roles/example-cert-admin"
```

## API 參考

### 角色端點（自訂 — 包含 `api_groups`）

角色始終位於 `system` 命名空間中。

| 方法 | 路徑 | 描述 |
| ------ | ---- | ----------- |
| POST | `/api/web/custom/namespaces/system/roles` | 建立自訂角色 |
| GET | `/api/web/custom/namespaces/system/roles` | 列出自訂角色 |
| GET | `/api/web/custom/namespaces/system/roles/{name}` | 取得自訂角色 |
| PUT | `/api/web/custom/namespaces/system/roles/{name}` | 取代自訂角色 |

### 角色端點（標準）

| 方法 | 路徑 | 描述 |
| ------ | ---- | ----------- |
| DELETE | `/api/web/namespaces/system/roles/{name}` | 刪除角色 |
| GET | `/api/web/namespaces/system/roles/{name}` | 取得角色 |
| GET | `/api/web/namespaces/system/roles` | 列出角色 |

### 探索端點（唯讀）

探索使用 `shared` 命名空間。

| 方法 | 路徑 | 描述 |
| ------ | ---- | ----------- |
| GET | `/api/web/namespaces/shared/api_groups` | 列出所有 api_groups |
| GET | `/api/web/namespaces/shared/api_group_elements` | 列出所有 api_group_elements |

### 憑證端點（角色授予存取權的對象）

將 `{namespace}` 替換為管理您憑證的命名空間。

| 方法 | 路徑 | 描述 |
| ------ | ---- | ----------- |
| POST | `/api/config/namespaces/{namespace}/certificates` | 建立憑證 |
| GET | `/api/config/namespaces/{namespace}/certificates` | 列出憑證 |
| GET | `/api/config/namespaces/{namespace}/certificates/{name}` | 取得憑證 |
| PUT | `/api/config/namespaces/{namespace}/certificates/{name}` | 取代憑證 |
| DELETE | `/api/config/namespaces/{namespace}/certificates/{name}` | 刪除憑證 |
| POST | `/api/config/namespaces/{namespace}/certificate_chains` | 建立憑證鏈 |
| GET | `/api/config/namespaces/{namespace}/certificate_chains` | 列出憑證鏈 |
| GET | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | 取得憑證鏈 |
| PUT | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | 取代憑證鏈 |
| DELETE | `/api/config/namespaces/{namespace}/certificate_chains/{name}` | 刪除憑證鏈 |

## 為何 `api_group` 和 `api_group_element` 是唯讀的

這些物件由 F5 XC **系統定義**。平台會為每個 API 路徑預先建立 `api_group_element` 項目，並將它們按功能區域分組為 `api_group` 物件。

您不需要建立這些物件 — 系統中每個資源的物件已經存在。您的工作是**探索**相關的群組名稱（步驟 1）並在您的自訂角色中**參考**它們（步驟 2）。

若要查看某個 api_group 涵蓋哪些個別 API 路徑，請列出 `api_group_elements` 端點並尋找與您的資源相關的元素：

```bash
curl -s \
  -H "Authorization: APIToken ${F5XC_API_TOKEN}" \
  "${F5XC_API_URL}/api/web/namespaces/shared/api_group_elements" \
  | jq '[.items[] | select(.name | test("ves-io-schema-certificate")) | .name]'
```
